# openclaw-audit -- Experiment Guide

A step-by-step test run of every aspect of the tool. All scan commands
write output to a single `scan_output/` folder in the project root.
This folder is listed in `.gitignore` so it never pollutes the repo.
You can delete it at any time and regenerate it by re-running a scan.

**What to observe:**

| Flag | Purpose |
|------|---------|
| `TARGET` | Required -- directory to scan |

| `--format` / `-f` | Output formats: `json`, `csv`, `mermaid` (default all three) |

| `--exclude` | Comma-separated directory names to skip |
| `--verbose` / `-v` | Show pipeline step logs |
| `--pretty` / `--no-pretty` | Indent JSON output (default: pretty) |

**Specific things to look for:**

- A TOKEN_STORAGE_RISK finding where `matched_text` says `<REDACTED>`
  -- the tool intentionally does not echo secrets into its own output
- A correlated finding whose `id` starts with `CORR-` (e.g. `CORR-ADMIN-*`)

## Experiment 8 -- Correlation Logic

**Goal:** Show the tool synthesizes cross-file insights, not just
single-file pattern matches.

First, regenerate full output:

```bash
python -m openclaw_audit scan tests/fixtures
```

### 8.1 -- List correlated findings

```bash
python -c "import json, pathlib; findings=json.loads(pathlib.Path('scan_output/findings.json').read_text())['findings']; corr=[f for f in findings if f['id'].startswith('CORR-')]; print(f'Correlated findings: {len(corr)}'); [(print(f'  {c[\"id\"]}: {c[\"title\"]}')) for c in corr]"
```

**Expected:** 3 correlated findings:

| ID prefix | Title | How it was built |
|-----------|-------|------------------|
| `CORR-ADMIN-*` | Publicly exposed admin / API with credential risk | `bind_address=0.0.0.0` + `admin_panel_port` + `api_key` in `.env` across multiple files |
| `CORR-PLUGIN-*` | Plugin loading with execution primitives | `plugin_dir` config + `subprocess.run` / `eval` in Python source |
| `CORR-SHARED-*` | Shared privileged agent with weak isolation | `shared_agent` with elevated capabilities + `shared_workspace: true` |

**Key point:** These are not direct regex matches. The correlator
noticed that multiple independent signals across different files combine
to form a higher-level risk.

## Experiment 9 -- Risk Scoring

### 9.1 -- The scoring formula

| Severity | Base score |
|----------|-----------|
| info | 1 |
| low | 3 |
| medium | 6 |
| high | 10 |
| critical | 15 |

Confidence multipliers: high = 1.0, medium = 0.75, low = 0.5.
Total is the sum of `(base * multiplier)` for all findings, clamped
to 0-100.

| Score range | Band |
|-------------|------|
| 0 -- 19 | Low |
| 20 -- 39 | Moderate |
| 40 -- 69 | High |
| 70 -- 100 | Critical |

### 9.2 -- Verify with unit tests

```bash
python -m pytest tests/test_risk.py -v
```

**What to observe:**

| Test | What it proves |
|------|----------------|
| `test_empty_findings` | Score is 0, band is "Low" |
| `test_single_critical` | 15 * 1.0 = 15, still "Low" (< 20) |
| `test_multiple_findings_accumulate` | 5 high findings = 50, "High" band |
| `test_confidence_reduces_score` | Same severity, low confidence scores less |
| `test_clamped_to_100` | 20 critical findings clamp to exactly 100, "Critical" |
| `test_category_breakdown` | Per-category subtotals are correct |

## Experiment 11 -- Code Quality & Extensibility

### 11.1 -- Project structure

```bash
dir openclaw_audit
dir openclaw_audit\rules
dir openclaw_audit\parsers
dir openclaw_audit\reporting
dir openclaw_audit\utils
```

**What to observe:**

- Clean modular layout: CLI, config, discovery, loaders, scanner,
  correlators, risk, models -- each in its own file
- Rules are in separate files by domain (network, auth, plugin,
  workspace, node, secret, execution)
- Parsers for each format (JSON, YAML, TOML, text)
- Reporters for each output (console, JSON, CSV, Mermaid)

### 11.2 -- Read a rule file

Open `openclaw_audit/rules/network_rules.py` and observe:

- Class-level metadata: `rule_id`, `severity`, `description`,
  `recommendation`
- The `apply()` method with two strategies: structured data search
  (recursive key matching) + text scan (line-by-line regex)
- Helper methods inherited from `BaseRule` (`_text_scan`, `_key_search`,
  `_make_finding`)

### 11.3 -- How to add a new rule

Adding a new detection rule is three steps:

1. Create a new file in `openclaw_audit/rules/` (e.g. `my_rules.py`)
2. Subclass `BaseRule`, set metadata, implement `apply()`
3. Register the class in `openclaw_audit/rules/__init__.py`

No changes to the scanner, correlator, or reporters are needed.
